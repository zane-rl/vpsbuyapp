# 系统设计

## 1. 技术选型

| 层 | 技术 | 说明 |
| --- | --- | --- |
| 框架 | Next.js 14（App Router, TypeScript） | 前后端一体，单进程部署 |
| UI | React 18 + Tailwind CSS | 服务端组件渲染列表，客户端组件处理表单交互 |
| 数据 | Prisma ORM + SQLite | 单文件数据库，便于备份迁移 |
| 鉴权 | 自实现 HMAC 签名 Cookie | 不引入额外鉴权依赖，兼容 Edge 中间件 |

选型理由：场景为小型内部运营工具，单人/小团队使用，数据量小。Next.js 一体化方案开发快、部署简单（一个 `npm run start`），SQLite 免去额外数据库运维。

## 2. 整体架构

```
浏览器
  ├── 首页 /               ──> 重定向到 /admin（需登录）
  ├── 公开页 /view         ──(服务端组件直接读库)──┐
  ├── 登录页 /login        ──> POST /api/auth/login │
  └── 后台 /admin/**       ──> /api/admin/**         │
                                                     ▼
                          middleware.ts (校验会话)  Prisma  ──> SQLite (dev.db / prod.db)
```

- **首页 `/`**：重定向到 `/admin`，未登录会被中间件转到 `/login`，不对外展示数据。
- **公开页 `/view`**：服务端组件（`src/app/view/page.tsx`）直接通过 Prisma 读库渲染，对金额脱敏，**不含后台入口**，可单独分享给客户。
- **管理页**：服务端组件渲染数据，表单交互由客户端组件通过 `fetch` 调用 `/api/admin/**`。
- **中间件**：`src/middleware.ts` 拦截 `/admin/**` 与 `/api/admin/**`，校验签名 Cookie；页面未授权重定向 `/login`，API 返回 401。

## 4. 鉴权设计（`src/lib/auth.ts`）

- **登录**：`POST /api/auth/login` 校验明文密码 `=== process.env.ADMIN_PASSWORD`（定长安全比较）。通过则用 `SESSION_SECRET` 对 payload 做 **HMAC-SHA256**，签名结果写入 httpOnly、SameSite=Lax 的 `vps_session` Cookie（有效期 7 天，生产环境 `secure`）。
- **验签**：中间件读取 Cookie，重新计算 HMAC 并定长比较。
- **退出**：`DELETE /api/auth/login` 将 Cookie `maxAge=0` 清除。
- **实现要点**：使用 **Web Crypto API**（`crypto.subtle`）而非 Node `crypto`，以便在 Next.js Edge 运行时（middleware）中可用，同时 Node 路由处理函数也兼容。
- 密码不入库，仅存环境变量；token 不含敏感信息，仅作"已认证"凭据。

## 5. 时间与状态计算（`src/lib/dates.ts`）

- `daysUntil(expiry)`：按自然日计算剩余天数（归零到当天 00:00，避免小时差偏差）。
- `expiryStatus(expiry)`：
  - `expired`（< 0 天）→ 红色徽章
  - `expiring`（0~7 天）→ 黄色徽章
  - `valid`（> 7 天）→ 绿色徽章
- 公开页与后台共用同一套工具，保证状态判断一致。

## 5.1 视觉风格与深色模式

- **设计基调**：现代简约 SaaS 风格，靛蓝（Indigo）主色，圆角卡片 + 柔和阴影 + 充足留白。
- **可复用样式**：在 `src/app/globals.css` 用 `@layer components` 集中定义 `.card` / `.input` / `.label` / `.btn-primary` / `.btn-success` / `.btn-secondary` / `.btn-danger` / `.btn-ghost` / `.badge` / `.table-head` / `.table-row`，每个都内置明暗两套配色。页面统一引用这些类，避免重复并保证一致。
- **深色模式**：Tailwind `darkMode: "class"`。
  - 防闪烁：`src/app/layout.tsx` 的 `<head>` 内联脚本在绘制前根据 `localStorage.theme` 或系统偏好给 `<html>` 加 `dark` 类。
  - 切换：`src/app/ThemeToggle.tsx`（客户端）切换 `dark` 类并写入 `localStorage`，放在后台导航、公开页、（登录页随系统）。
- **状态色**：`src/lib/dates.ts` 的 `statusBadgeClass` / `statusDotClass` 提供含深色变体的徽章与圆点颜色（绿/黄/红 = 有效/即将到期/已过期）。
- **导航布局**：后台保持顶部导航（粘性 + 毛玻璃背景）；公开页 `/view` 为单页卡片栅格。

## 6. 关键设计决策

- **不暴露全部客户**：不提供“展示所有客户”的全局公开页；裸 `/view` 重定向到登录，公开仅限 `/view/<客户ID>`（只读该客户数据）。客户专属页只取 `enabled: true` 的节点。
- **续费事务**：新增续费记录与更新 VPS 到期时间在 `prisma.$transaction` 中原子完成，避免数据不一致。
- **服务端/客户端边界**：纯函数 `toFormData` 单独放在 `vpsFormData.ts`（非 `"use client"`），避免被服务端组件当作客户端引用代理而无法调用。
- **财务汇总**：用 Prisma `aggregate` 分别对 VPS 表与续费表求和后相加，避免全量加载到内存。
- **复制 VPS**：不新增专用接口，复用新增流程——列表「复制」跳转 `/admin/vps/new?from=<id>`，新增页（服务端组件）读取该 VPS 并经 `toFormData` 预填表单（清空 id/IP、名称加"-副本"），用户修改后走普通 POST 创建。
- **删除续费记录**：`DELETE /api/admin/renewals/{id}` 仅删历史记录，刻意不联动修改 VPS 的 `expiryDate`，避免删除历史误改当前有效期；到期时间由编辑 VPS 显式控制。
